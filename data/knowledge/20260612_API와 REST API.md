# API와 REST API — 프로그램끼리 대화하는 창구

## 🪟 API가 뭘까 — 식당의 '주문 창구'

**API(Application Programming Interface)**는 "프로그램이 다른 프로그램에게 무언가를 요청하는 **정해진 창구**"입니다.

**비유:** 식당의 주문 창구. 손님은 주방에 직접 들어가지 않습니다. **메뉴판(정해진 형식)**으로 주문 창구에 시키면, 주방이 만들어 내보내요. API는 바로 이 **'정해진 주문 창구 + 메뉴판'** 입니다.

예: 날씨 앱은 기상청 컴퓨터 속으로 직접 못 들어갑니다. 대신 기상청이 열어둔 **날씨 API**라는 창구에 "서울 날씨 줘"라고 요청해서 받아옵니다.

---

## 🔌 왜 API가 중요할까

API 덕분에 **서로 다른 서비스가 연결**됩니다.
- 쇼핑몰에 '카카오로 로그인' → 카카오 API
- 식당 앱의 지도 → 구글 지도 API
- 우리가 쓰는 AI 도구 → AI 회사의 API

**비유:** 콘센트(표준 규격). 제조사가 달라도 플러그를 꽂으면 전기가 통하듯, API라는 표준 창구가 있어 서로 다른 회사 서비스가 술술 연결됩니다.

---

## 🌐 REST API — 가장 흔한 API 방식

**REST API**는 API를 만드는 여러 방식 중 **가장 널리 쓰이는 약속**입니다. 웹 주소(URL)와 정해진 '동작 단어'로 요청해요.

| 동작 단어 | 뜻 | 식당 비유 |
|---|---|---|
| **GET** | 가져오기 | "메뉴 보여줘" |
| **POST** | 새로 만들기 | "주문 넣을게" |
| **PUT** | 수정하기 | "주문 바꿀게" |
| **DELETE** | 삭제하기 | "주문 취소" |

예: `GET /weather/seoul` = "서울 날씨를 **가져와줘**". 주소만 봐도 무슨 요청인지 읽힙니다 — 그게 REST의 장점이에요.

---

## 📦 주고받는 데이터는 보통 JSON

API가 데이터를 주고받을 때 흔히 **JSON**이라는 형식을 씁니다. `이름: 값` 짝으로 적은 **깔끔한 메모**라고 보면 돼요.

```json
{ "도시": "서울", "날씨": "맑음", "기온": 24 }
```

사람도 읽을 수 있고 컴퓨터도 읽기 쉬워서, API 세계의 **공용 서식**입니다.

---

## 💡 핵심 교훈

- **API** = 프로그램끼리 주고받는 **정해진 주문 창구**. 콘센트처럼 서로 다른 서비스를 연결합니다.
- **REST API** = 가장 흔한 방식. **GET(가져오기)/POST(만들기)/PUT(수정)/DELETE(삭제)** 4동작으로 요청.
- 데이터는 주로 **JSON**(이름:값 메모)으로 오갑니다. "AI도, 자동화도 결국 API로 연결된다"는 감만 잡으면 충분합니다.
